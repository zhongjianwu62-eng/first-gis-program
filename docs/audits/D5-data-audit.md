## 与 D3/D4 的一致性检查

D3 确定“离线 Pipeline + PostGIS + FastAPI + Vue 3 GIS 工作台”的架构。D4 将其拆分为 `frontend`、`backend`、`pipeline`、`database`、`deploy`、`tests` 和 `docs`。

D5 的数据方案进一步明确：

1. Pipeline 输入为行政边界和 OSM 路网；
2. Pipeline 在 EPSG:32650 下进行米制计算；
3. PostGIS 存储 EPSG:4326 正式几何和米制结果字段；
4. 前端底图通过配置层接入，不参与指标计算；
5. 后端查询已发布数据版本，不现场重算指标；
6. 数据质量审计输出作为后续 Pipeline 验收依据。

未发现与 D3/D4 冲突的设计。

## 数据风险分级

| 风险 | 等级 | 处理方式 |
|---|---|---|
| 思明区边界缺少鼓浪屿 | Blocker | 必须修复后才能进入正式版本 |
| 道路几何大量为空 | Blocker | 重新获取或修复数据 |
| 双路网筛选规则失效 | Blocker | 修正筛选规则并重新审计 |
| 坐标系统错误导致米制计算失真 | Blocker | 修正 CRS 流程并重算 |
| 真实底图凭证进入 GitHub | Blocker | 立即移除、轮换凭证并记录事故 |
| 道路名称缺失比例较高 | Warning | 记录比例，说明搜索功能边界 |
| service 道路占比偏高 | Warning | 检查是否需要细分过滤 |
| OSM 与天地图轻微视觉偏移 | Warning | 记录偏移风险，不私自改坐标 |
| 鼓浪屿是独立网络子图 | Info | 作为真实地理结构解释 |
| 边界处无道路网格 | Info | 标记边界网格并解释 |

## 审计修正记录

1. 将底图选择与天地图凭证管理作为 D5 正式审计内容，而非仅作为前端配置问题。
2. 明确“底图不等于分析数据源”，避免混用天地图底图和 OSM 分析数据。
3. 将鼓浪屿真实独立路网作为正常地理结构记录。
4. 将 EPSG:32650 确认为首版米制计算候选投影，并保留正式数据阶段复核要求。
5. 将凭证安全纳入 Blocker 风险，防止真实凭证进入公开仓库。

## 审计结论

D5 数据方案完整、内部一致，并与 D1 项目章程、D2 需求分析、D3 总体架构和 D4 模块设计保持一致。D5 明确了研究范围、边界来源策略、底图与凭证管理、OSM 路网数据来源、双路网筛选、坐标系统和数据质量审计规则。

D5 可以提交到功能分支并进入线上 PR 审阅流程。合并后可作为 D6 工程骨架搭建、D7 Pipeline 开发和后续 PostGIS 入库的输入基线。
