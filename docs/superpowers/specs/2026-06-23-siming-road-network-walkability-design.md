## 1. 项目定位

本项目建设一套面向城市规划人员和交通研究者的专业 GIS 分析工作台。平台以城市路网结构分析和路网结构层面的可步行性潜力评价为核心，识别厦门市思明区的重要道路、路网连接特征、步行优势区域和潜在薄弱区域。

项目不止制作一张静态专题地图，而是形成一套可以查询、筛选、比较并解释分析结果的完整系统。开发过程覆盖数据获取与治理、空间分析、PostGIS 数据管理、FastAPI 服务、Vue 3 可视化、测试、部署和工程文档。

### 1.1 核心研究问题

1. 哪些路段在思明区路网中具有较高的结构中心性？
2. 哪些区域具有较高的路口密度、连通性和步行道路覆盖程度？
3. 哪些区域存在路网结构或步行网络方面的潜在短板？
4. 路网中心性较高的区域是否同时具备较高的可步行性潜力？
5. 不同路网形态在厦门岛主体与鼓浪屿之间表现出哪些差异？

### 1.2 研究范围

- 行政范围完整覆盖厦门市思明区，并包含鼓浪屿。
- 厦门岛主体和鼓浪屿分别保持真实的网络连通关系，不建立虚假的跨海道路连接。
- 现有北京中心城区道路数据仅作为早期技术验证样本，不作为最终分析成果。
- 最终数据应基于可复现的 OpenStreetMap/Overpass 或 OSMnx 获取流程生成。

### 1.3 分析尺度

平台采用双尺度表达：

- **道路级尺度**：计算和展示道路属性、长度、中心性及相关结构指标，用于解释具体道路问题。
- **500 米网格尺度**：汇总路口密度、路段特征、连通性、步行道路覆盖和道路等级影响，用于比较区域差异。

### 1.4 首版评价范围

首版综合可步行性评价包含：

1. 路口密度；
2. 路段长度特征；
3. 路网连通性；
4. 步行道路覆盖程度；
5. 道路等级对步行环境的影响。

首版评价结果应表述为“路网结构层面的可步行性潜力”，不能替代对人行道质量、过街安全、坡度、遮阴、街道活力和设施可达性的完整评价。

首版暂不引入 POI、公交站、绿地和公共服务设施可达性。这些内容作为后续研究型版本的扩展方向。

## 2. 产品目标与成功标准

### 2.1 产品目标

平台应帮助专业用户完成以下分析链路：

> 打开平台 → 选择路网或可步行性指标 → 观察空间分布 → 筛选高值或低值区域 → 查询道路或网格详情 → 阅读指标构成与评分解释 → 形成可用于规划研究的判断。

### 2.2 质量目标

项目追求研究、工程和视觉三方面均衡：

- **研究可信**：指标定义明确，计算过程可复现，评价边界不过度解释。
- **工程完整**：前端、后端、空间数据库、数据流水线、测试和部署链路完整。
- **视觉专业**：地图信息层级清晰，色彩语义一致，交互符合专业 GIS 工作场景。

### 2.3 首版完成判定

在不临时修改代码的情况下，系统能够连续完成以下操作：

> 启动系统 → 加载思明区数据 → 切换路网指标 → 查看 500 米网格评价 → 筛选低值区域 → 点选具体道路或网格 → 阅读指标解释 → 查看统计图表。

全过程应无阻断性错误、无错误的跨海网络连接，并能够解释所有核心指标的来源、方向和含义。

## 3. 用户与使用场景

### 3.1 主要用户

- 城市规划人员；
- 交通规划与城市空间研究者；
- 需要开展路网结构分析的 GIS 专业人员。

### 3.2 典型场景

1. 规划人员快速识别路网中心性较高的关键道路。
2. 研究者比较不同 500 米网格的路网结构和可步行性潜力。
3. 用户筛选综合得分偏低区域，并查看其单项指标短板。
4. 用户比较厦门岛主体与鼓浪屿的网络结构差异。
5. 用户点选道路或网格，获得可解释的属性、指标和评分信息。

## 4. 功能范围

### 4.1 首版必须完成

#### 4.1.1 地图浏览

- 展示思明区行政边界、完整分析路网和 500 米分析网格；
- 支持地图缩放、平移、范围复位和底图切换；
- 正确表达鼓浪屿的独立路网及其空间位置；
- 提供加载状态、空数据状态和异常提示。

#### 4.1.2 路网结构分析

- 按道路等级分类着色；
- 展示至少一种适用于路段的中心性指标；
- 支持道路等级和中心性指标切换；
- 支持道路点选与高亮；
- 展示道路名称、道路类型、长度、连通属性和中心性结果。

#### 4.1.3 可步行性分析

- 计算首版约定的五类指标；
- 明确每个指标的计算公式、统计方向和适用边界；
- 对指标执行可解释的标准化；
- 形成 500 米网格综合得分；
- 支持单项指标与综合评价结果切换；
- 使用分级设色表达空间差异。

#### 4.1.4 空间查询与筛选

- 按道路等级筛选；
- 按中心性等级筛选；
- 按可步行性等级或分值范围筛选；
- 点选网格查看综合得分和各单项指标；
- 地图、统计信息和详情面板保持联动。

#### 4.1.5 统计与解释

- 展示分析范围、道路数量、路网总长度等概览指标；
- 展示核心指标分布图表；
- 列出典型高值和低值区域；
- 提供指标定义、评分方向和结果解释；
- 避免将相关性表述为因果关系。

#### 4.1.6 数据与工程链路

- 数据获取、清洗、建图和指标计算可重复执行；
- 道路、节点、网格和分析结果存入 PostGIS；
- FastAPI 提供稳定、文档化的查询接口；
- Vue 3 前端通过接口获取数据；
- 提供本地环境、数据库初始化、数据导入和系统启动说明；
- 建立可公开访问的部署版本。

### 4.2 有余力再做

- 两个区域或网格的并排比较；
- 分析结果导出为 CSV 或 GeoJSON；
- 地图截图或报告摘要导出；
- 夜间主题；
- 可分享的查询参数与地图状态；
- 综合权重的交互式敏感性比较。

这些功能只能在首版全部验收通过后进入开发。

### 4.3 首版明确不做

- POI、公交站、绿地和公共服务设施可达性；
- 步行路径规划和导航；
- 实时交通、人流或手机信令分析；
- 用户登录、角色权限和复杂后台管理；
- 任意城市的在线实时建图与指标计算；
- AI 自动生成规划决策；
- 商业级高并发基础设施的完整落地。

## 5. 总体技术方向

### 5.1 系统形态

系统采用专业 GIS 分析工作台：

- 中央区域为主要地图；
- 左侧承载图层、指标和筛选控制；
- 右侧承载统计结果、指标解释和道路/网格详情；
- 关键图表与地图保持交互联动。

具体页面布局、视觉规范和组件结构在后续详细设计阶段确定。

### 5.2 三层架构

1. **数据层**：PostgreSQL/PostGIS 保存行政边界、路网、节点、500 米网格、指标和数据版本信息。
2. **服务层**：FastAPI 提供范围查询、属性查询、统计和健康检查接口。
3. **表现层**：Vue 3 与地图引擎构建专业 GIS 工作台，负责地图表达、筛选、查询和图表联动。

数据处理流程作为独立模块运行，负责下载、清洗、拓扑处理、指标计算和入库，不由浏览器实时执行重型分析。

### 5.3 双链路迭代

- **产品闭环链路**：数据入库 → API 查询 → 前端地图 → 用户交互 → 结果解释。
- **质量增强链路**：算法验证 → 工程规范 → 性能优化 → 视觉打磨 → 测试与部署。

每个阶段先保证产品闭环继续可运行，再提升对应质量，避免在开发末期才发现数据、接口或部署链路无法打通。

## 6. 验收与质量标准

### 6.1 研究与数据质量

- 研究范围完整覆盖思明区及鼓浪屿；
- 记录数据来源、获取时间、查询范围、坐标系和字段定义；
- 数据下载、清洗、拓扑处理、指标计算和入库过程可重复执行；
- 检查重复道路、无效几何、异常短路段和异常连通分量；
- 保留并正确处理 `bridge`、`tunnel`、`layer` 等可能影响拓扑的字段；
- 不产生厦门岛主体与鼓浪屿之间的虚假路网连接；
- 所有指标均有定义、公式、方向和解释边界；
- 标准化方法和综合权重公开；
- 用户可以分别查看单项指标，不能只得到一个无法解释的综合分数。

### 6.2 功能质量

- 道路等级、中心性和可步行性图层显示正确；
- 道路与 500 米网格均可点选查询；
- 筛选、指标切换、图表和地图联动有效；
- 加载失败、无结果和接口异常有明确反馈；
- 核心分析链路不存在阻断性错误。

### 6.3 性能基线

- 普通教学电脑与稳定网络环境下，首屏主体内容目标在 3 秒内出现；
- 常用空间查询目标响应时间不超过 1 秒；
- 指标切换和筛选无明显长时间卡顿；
- 地图缩放和平移保持可用和流畅；
- 不向浏览器无控制地传输完整大型 GeoJSON；
- 性能测试记录环境、数据规模和测量方式。

以上为课程项目的性能目标，不等同于商业服务等级协议或高并发承诺。

### 6.4 界面与易用性

- 地图是工作台的主要视觉区域；
- 颜色含义稳定，图例、地图和详情面板保持一致；
- 高低值配色兼顾辨识度和色觉友好；
- 重要操作无需阅读长篇说明即可完成；
- 主流桌面浏览器可用；
- 1366×768 分辨率下核心功能不被遮挡；
- 平板和手机仅保证基础查看，不作为首版重点。

### 6.5 工程质量

- Vue 3、FastAPI、PostGIS 和数据处理模块职责分离；
- 前端组件、接口客户端、后端路由、业务服务和数据访问边界清晰；
- 密钥、数据库密码和部署参数通过环境变量管理；
- 主分支始终保持可运行；
- 功能采用独立分支、小步提交和评审式自检；
- API 具有统一响应与错误处理策略；
- 提供数据库初始化、数据导入和本地启动说明；
- 至少覆盖核心算法测试、接口测试和关键界面流程测试。

### 6.6 部署与交付

最终交付包括：

1. 可运行的 Vue 3 前端；
2. FastAPI 服务；
3. PostGIS 数据库结构与初始化方式；
4. 可复现的数据获取和处理脚本；
5. 需求分析与系统设计报告；
6. 数据说明和数据字典；
7. 项目 README 与部署文档；
8. 测试记录；
9. 答辩 PPT 与演示脚本；
10. 可公开访问的部署版本；
11. 清晰、连续的版本历史。

## 7. 商业化演进约束

首版不建设完整的高并发基础设施，但架构不得阻塞后续商业化演进。

### 7.1 首版预留

- API 使用稳定、清晰的契约；
- FastAPI 服务保持无状态；
- 数据表包含必要的空间索引、普通索引和数据版本字段；
- 重型指标采用预计算，不在普通请求中重复计算；
- 数据查询按范围和用途返回，避免无边界全量传输；
- 前端数据访问集中封装，允许后续替换 GeoJSON 查询方式；
- 配置与密钥外置，支持多环境部署。

### 7.2 后续升级顺序

1. 将大规模地图数据转换为 MVT 矢量瓦片；
2. 使用对象存储和 CDN 分发静态资源与瓦片；
3. 使用 Redis 缓存热门查询和统计结果；
4. 使用 PgBouncer 等连接池控制数据库连接；
5. 将 FastAPI 部署为多进程、多实例的无状态服务；
6. 通过负载均衡和自动伸缩扩展 API；
7. 为耗时计算引入异步任务队列；
8. 根据容量需求建设数据库只读副本、分区和高可用方案；
9. 增加认证、租户隔离、限流、配额和计费；
10. 建设日志、指标、链路追踪、告警、压测、灰度发布和回滚能力。

升级时优先遵循：

> 预计算 → 空间索引 → 矢量瓦片 → CDN/缓存 → 无状态 API → 水平扩容。

## 8. 风险与应对

| 风险 | 影响 | 应对策略 |
|---|---|---|
| OSM 步行道路或属性缺失 | 指标存在偏差 | 固定数据版本，执行完整性检查，记录缺失，不伪造未知属性 |
| 路网拓扑错误 | 中心性和连通性失真 | 保留高架、隧道、层级字段，检查连通分量并人工抽查关键区域 |
| 鼓浪屿错误接入厦门岛网络 | 产生虚假可达关系 | 按真实几何和拓扑建图，验证不同连通分量，不以直线距离强制连接 |
| 可步行性概念被过度解释 | 研究结论失真 | 始终使用“路网结构层面的可步行性潜力”，公开未纳入因素 |
| 综合权重具有主观性 | 排名随权重变化 | 公开权重和标准化方式，保留单项指标，后续增加敏感性分析 |
| 地图数据量导致卡顿 | 影响演示和使用 | 范围查询、空间索引、几何简化、分级加载，必要时升级矢量瓦片 |
| 前后端与算法范围过大 | 20 天内无法完整交付 | 先完成最小闭环，扩展功能不得阻塞核心系统 |
| 单人承担全部角色 | 缺少并行和同伴评审 | 按角色分阶段工作，保存决策与测试记录，执行设计和代码自审 |
| 环境和密钥泄露 | 安全与部署风险 | 使用环境变量、示例配置和提交前敏感信息检查 |
| 线上环境与本地不一致 | 部署失败 | 固定依赖版本，提供初始化脚本和环境检查，尽早执行部署验证 |

## 10. 每日工作制度

每天按 4–6 小时安排：

1. **计划（15 分钟）**：明确当天唯一主要目标、任务范围和验收条件。
2. **设计或学习（45–60 分钟）**：先理解原理、数据和方案，再实施。
3. **核心实施（2–3 小时）**：完成一个可运行、可查看或可验证的小增量。
4. **测试与打磨（45–60 分钟）**：检查正确性、异常状态、性能和视觉效果。
5. **记录与提交（30 分钟）**：更新文档、问题日志、变更记录并小步提交。
6. **日终复盘（15 分钟）**：记录完成项、问题、决策、技术债和次日入口。

### 10.1 每日完成定义

当天任务同时满足以下条件才算完成：

- 产物可以打开、运行或检查；
- 达到当天约定的验收条件；
- 已处理已知阻断性错误；
- 数据、代码或结论来源明确；
- 文档与实际结果一致；
- 有清晰的版本记录；
- 次日有明确的继续入口。

### 10.2 变更控制

新增需求先进入候选清单，不立即插入开发。只有同时满足以下条件，才可进入首版：

- 直接服务核心研究问题；
- 不破坏 20 天进度；
- 能定义明确验收标准；
- 收益高于新增复杂度。

否则进入后续版本候选清单。
