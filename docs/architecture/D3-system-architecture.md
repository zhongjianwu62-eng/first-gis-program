## 1. 架构目标

本项目建设一套面向城市规划人员、交通研究者和 GIS 分析人员的路网结构与可步行性分析工作台。总体架构需要同时满足课程设计交付、个人作品集展示和后续商业化演进的要求。

D3 的目标不是立即写代码，而是形成可执行的总体架构蓝图，用于指导 D4 模块拆分、D5 数据源审计、D6 工程脚手架搭建和后续开发。

总体架构遵循以下原则：

1. **离线计算，在线查询**：中心性、网格指标、综合分等重型 GIS 计算由离线 Pipeline 完成，在线 API 只负责查询已发布结果。
2. **前端表达，后端封装，数据库治理**：前端不直接访问数据库，后端不承担重型分析，PostGIS 不暴露给浏览器。
3. **数据版本可追踪**：道路、节点、网格和指标结果均绑定数据版本，保证结果可复现、可审计。
4. **首版克制，边界预留**：首版不引入 Redis、微服务、Kubernetes、实时交通和在线权重编辑，但保留未来演进空间。
5. **企业级留痕**：每日形成工作日志、审计记录、Git 提交和线上合并记录。

## 2. 总体架构

系统采用“Vue 3 GIS 分析工作台 + FastAPI 只读查询服务 + PostGIS 空间数据库 + 独立离线 Pipeline”的模块化单体架构。

```text
Vue 3 GIS 分析工作台
MapLibre · Pinia · ECharts
        ↓ /api/v1
FastAPI 只读查询服务
router → service → repository → database
        ↓ SQL / PostGIS
PostgreSQL + PostGIS
道路 · 节点 · 网格 · 指标 · 数据版本
        ↑ 发布数据版本
独立离线 Pipeline
下载 · 清洗 · 建图 · 计算 · 入库
```

该架构将系统分为四个清晰模块：

| 模块 | 主要职责 | 不承担的职责 |
|---|---|---|
| 前端 GIS 工作台 | 地图交互、指标切换、图表展示、详情解释、CSV 导出入口 | 不直接连接数据库，不执行重型 GIS 计算 |
| FastAPI 查询服务 | 参数校验、业务规则、只读查询、错误处理、日志记录 | 不下载 OSM 数据，不现场计算中心性和综合指标 |
| PostGIS 数据库 | 空间数据存储、空间索引、范围查询、指标结果管理、数据版本管理 | 不处理页面交互，不管理前端状态 |
| 离线 Pipeline | 数据获取、清洗、建图、指标计算、版本发布 | 不响应网页请求，不暴露公网接口 |

## 3. 技术选型

### 3.1 前端

前端采用：

- Vue 3；
- TypeScript；
- Composition API；
- Pinia；
- MapLibre GL JS；
- Element Plus；
- ECharts。

Vue 3 与 TypeScript 用于保证开发效率和类型可靠性。Pinia 负责管理地图状态、图层状态、指标状态、选中对象和查询状态。MapLibre GL JS 负责地图渲染，并为后续矢量瓦片升级保留空间。Element Plus 提供基础控件，ECharts 负责指标卡、柱状图、雷达图和排行榜等分析图表。

首版不选择 Leaflet 作为主地图引擎，因为本项目需要网格图层、道路图层、分级设色、未来矢量瓦片和较强地图样式能力，MapLibre 更适合后续演进。

### 3.2 后端

后端采用：

- FastAPI；
- Pydantic；
- SQLAlchemy 2.0；
- GeoAlchemy2；
- Alembic。

FastAPI 用于构建清晰的只读 API。Pydantic 用于请求参数和响应结构校验。SQLAlchemy 2.0 与 GeoAlchemy2 用于数据库访问和空间字段处理。复杂空间查询允许使用参数化显式 SQL，避免为了形式化 ORM 牺牲 GIS 查询性能。Alembic 用于管理数据库结构迁移。

### 3.3 数据库

数据库采用 PostgreSQL + PostGIS。PostGIS 作为正式空间数据中心，负责道路、节点、500 米网格、指标结果、指标定义和数据版本管理。

GeoJSON 可作为接口返回格式，但不作为正式主存储方式。正式业务数据必须进入 PostGIS，以支持空间索引、范围查询、版本追踪和后续性能优化。

### 3.4 离线 Pipeline

Pipeline 采用 Python 技术栈，后续可使用 OSMnx、GeoPandas、NetworkX、Shapely 和 PyProj 等库。具体版本和数据源细节在 D5 数据源审计和 D6 工程搭建时确定。

Pipeline 负责下载或导入思明区道路数据、统一坐标系统、清洗道路类型、构建机动车路网与步行路网、生成 500 米网格、计算路网结构指标和可步行性综合指标，并将结果写入 PostGIS。

### 3.5 工程化与部署

本地开发采用 Docker Compose 管理 PostGIS 和基础服务，前端使用 Vite 本地热更新，后端和 Pipeline 可独立运行。生产首版采用单台云服务器、Docker Compose、Nginx 和 HTTPS。

GitHub Actions 用于格式检查、类型检查、单元测试、构建检查、敏感信息扫描和后续自动审计。

## 4. 系统数据流与模块边界

### 4.1 总体数据流

```text
OSM / 行政区 / 原始道路数据
        ↓
离线 Pipeline
        ↓
清洗后的路网、节点、网格、指标
        ↓
PostgreSQL + PostGIS
        ↓
FastAPI 查询服务
        ↓
Vue 3 GIS 分析工作台
        ↓
地图、图表、指标卡、详情面板、CSV 导出
```

系统采用“离线计算、在线查询、前端表达”的数据流。离线 Pipeline 负责道路数据清洗、路网构建和指标计算；PostGIS 负责空间数据存储和范围查询；FastAPI 负责只读 API 和业务规则封装；Vue 3 前端负责地图交互和分析结果表达。

### 4.2 在线请求流

用户打开系统后，前端先获取系统配置、指标定义和当前数据版本。用户移动地图或调整筛选条件时，前端按当前地图视野 bbox 请求道路或网格数据。用户点击网格或道路时，前端再请求对应详情。

在线请求不应触发重型指标计算。所有正式展示结果均来自已发布数据版本。

### 4.3 离线处理流

离线 Pipeline 的处理流为：

1. 确定研究范围：厦门市思明区；
2. 获取道路和行政区数据；
3. 数据清洗与坐标统一；
4. 构建机动车路网和步行路网；
5. 生成 500 米分析网格；
6. 计算路网结构指标；
7. 计算可步行性综合指标；
8. 写入 PostGIS；
9. 生成数据版本记录与处理报告。

如果 Pipeline 失败，只记录失败状态，不替换当前线上数据版本。

## 5. API 与数据访问总体设计

API 采用 REST 风格、`/api/v1` 版本前缀和只读查询接口。

### 5.1 API 分组

| 分组 | 作用 | 示例 |
|---|---|---|
| 系统接口 | 健康检查、版本信息 | `/health`、`/api/v1/meta` |
| 网格接口 | 查询 500 米网格和网格详情 | `/api/v1/grids`、`/api/v1/grids/{id}` |
| 道路接口 | 查询道路、道路详情、道路搜索 | `/api/v1/roads`、`/api/v1/roads/search` |
| 指标接口 | 查询指标说明、指标统计、排行榜 | `/api/v1/indicators`、`/api/v1/rankings` |
| 统计接口 | 查询全区概览和分布统计 | `/api/v1/statistics/overview` |
| 导出接口 | 导出当前筛选结果 | `/api/v1/export/grids.csv` |

### 5.2 地图数据查询

道路和网格查询必须支持 bbox 范围过滤，避免一次性加载全区详细几何。

示例：

```text
GET /api/v1/grids?bbox=118.05,24.42,118.18,24.52&indicator=walkability_score
GET /api/v1/roads?bbox=118.05,24.42,118.18,24.52&road_type=walkable
```

地图几何类数据返回 GeoJSON，统计、详情和指标说明返回普通 JSON。首版不实现 MVT，但 API 边界为后续升级保留空间。

### 5.3 详情、搜索、排行榜与导出

网格详情接口返回网格编号、综合分、单项指标原始值、标准化值、全区均值对比、百分位、数据版本和指标解释。

道路搜索接口支持名称模糊搜索，返回道路名称、道路编号、道路类型、中心点和可选简化几何，用于前端定位和高亮。

排行榜接口支持指标选择、高值或低值排序、数量限制和可选空间范围过滤。

CSV 导出接口只导出当前筛选结果，导出字段包含网格编号、中心点坐标、综合分、单项指标、百分位、数据版本和导出时间。

### 5.4 错误结构

所有 API 错误统一返回：

```json
{
  "error_code": "INVALID_BBOX",
  "message": "地图范围参数不正确，请重新调整地图范围。",
  "request_id": "req_20260623_xxxx"
}
```

错误响应不暴露数据库错误、代码堆栈、服务器路径或内部凭证信息。

### 5.5 后端分层

后端内部采用：

```text
router → service → repository → database
```

router 接收请求和返回响应；service 处理业务规则；repository 封装数据库查询；database 管理数据库连接。复杂空间查询集中放在 repository 层，前端不感知数据库表结构。

## 6. 数据库与数据版本总体设计

PostgreSQL + PostGIS 是系统正式空间数据中心。核心数据表包括：

| 表 | 作用 |
|---|---|
| `data_versions` | 记录每次正式数据处理版本 |
| `roads` | 存储道路几何、名称、道路类型 |
| `road_nodes` | 存储道路网络节点 |
| `analysis_grids` | 存储 500 米分析网格 |
| `grid_indicators` | 存储每个网格的指标结果 |
| `road_indicators` | 存储道路级指标结果 |
| `indicator_definitions` | 存储指标名称、解释、单位、方向和权重 |
| `pipeline_runs` | 记录每次离线处理运行情况 |

### 6.1 数据版本

`data_versions` 记录数据版本号、研究区域、数据来源、OSM 数据获取时间、处理脚本版本、指标算法版本、是否当前启用、创建时间和备注说明。

示例版本号：

```text
siming-osm-2026-06-v1
```

道路、节点、网格和指标结果均绑定数据版本。系统始终读取当前启用版本。

### 6.2 道路、节点和网格

`roads` 存储清洗后的道路几何、名称、类型、是否可步行、是否机动车可通行、长度和来源 ID。道路数据需要支持机动车路网和步行路网两种分析视角。

`road_nodes` 存储路网节点、节点类型、连接道路数量、是否交叉口、几何点和网络类型。

`analysis_grids` 存储 500 米分析网格、中心点、所属研究区域和是否参与统计。边界处不完整网格可通过字段控制是否纳入统计。

### 6.3 指标结果和指标定义

`grid_indicators` 保留每个网格的原始指标值、标准化值、综合分、百分位和分级等级。原始值和标准化值都必须保留，以保证结果可解释。

`indicator_definitions` 记录指标编码、中文名、英文名、解释、单位、指标方向、权重、是否参与综合评分和展示顺序。指标定义不写死在前端。

### 6.4 Pipeline 运行记录

`pipeline_runs` 记录运行 ID、数据版本 ID、开始时间、结束时间、运行状态、输入来源、处理脚本版本、日志摘要、错误信息和生成记录数量。

只有完整跑通、通过校验、生成审计记录的数据版本，才允许成为当前启用版本。旧版本不立即删除。

### 6.5 索引策略

首版至少建立道路几何空间索引、网格几何空间索引、节点几何空间索引、数据版本索引、道路名称索引和指标分数索引。

## 7. 前端 GIS 工作台总体设计

前端采用 GIS 分析工作台布局，由顶部全局栏、左侧控制面板、地图主视图和右侧详情面板组成。

```text
┌──────────────────────────────────────────────┐
│ 顶部栏：项目名称 / 数据版本 / 导出 / 帮助说明 │
├──────────────┬───────────────────────┬───────┤
│ 左侧控制面板 │        地图主视图        │ 右侧详情 │
│ 指标选择     │        网格/道路图层     │ 指标解释 │
│ 图层开关     │        图例             │ 网格详情 │
│ 筛选条件     │        搜索定位          │ 排行榜   │
└──────────────┴───────────────────────┴───────┘
```

### 7.1 顶部栏

顶部栏展示系统名称、当前研究区、当前数据版本、导出入口、指标说明入口和系统状态提示。

### 7.2 左侧控制面板

左侧控制面板提供指标选择、图层开关、道路类型筛选、分级方式、排序方式和筛选条件重置。首版展示固定公开权重，但不提供在线权重编辑。

### 7.3 地图主视图

地图主视图展示思明区边界、500 米分析网格、道路网络、当前选中网格、搜索道路高亮、图例、比例尺、缩放控件和底图切换。

地图支持平移缩放、点击网格查看详情、点击道路查看道路信息、道路搜索定位、指标切换设色和道路类型样式切换。

### 7.4 右侧详情面板

默认状态展示当前指标解释、全区概览、高值网格榜单和低值网格榜单。点击网格后展示网格编号、综合分、百分位、全区均值对比、原始指标值、标准化指标值、雷达图和解释文本。

### 7.5 图表、搜索与导出

首版图表包括指标卡、柱状图、雷达图和排行榜。道路搜索聚焦道路名称模糊搜索。CSV 导出当前筛选结果，文件名包含研究区、指标和日期。

### 7.6 前端状态和错误处理

前端状态分为地图状态、图层状态、指标状态、选择状态、查询状态和系统状态，由 Pinia 统一管理。

前端错误提示面向普通用户，不展示数据库错误或接口堆栈。常见错误包括地图数据加载失败、搜索无结果、当前视野数据量过大、后端服务暂不可用和 CSV 导出失败。

## 8. 部署、运维与高并发演进

### 8.1 首版部署

首版采用单台云服务器 + Docker Compose + Nginx + HTTPS 的轻量生产架构。

```text
用户浏览器
    ↓ HTTPS
Nginx
    ├── 前端静态文件
    └── FastAPI API 服务
            ↓
        PostGIS 数据库
            ↑
        离线 Pipeline
```

用户只访问 Nginx。前端由 Nginx 提供，API 请求转发给 FastAPI。Pipeline 不暴露公网，只在后台离线运行。

### 8.2 环境与配置

系统保留 local、staging 和 production 三类环境概念。课程项目阶段可以先实现 local 和 production。

所有关键配置通过 `.env.example` 说明，真实 `.env` 不提交 GitHub。配置包括数据库地址、API 地址、底图 key、CORS 域名、日志级别、当前环境和数据版本策略。

### 8.3 日志、健康检查与审计

后端采用 JSON 结构化日志，包含时间、request id、请求路径、状态码、耗时、错误码和环境信息。健康检查接口为 `/health`。

GitHub Actions 负责前端格式检查、类型检查、构建检查、后端测试、后端代码检查、敏感信息扫描和文档基础检查。

### 8.4 首版性能策略

首版性能策略包括 bbox 查询、空间索引、离线预计算、查询范围限制、导出数量限制、前端避免一次性加载全区详细几何、常用统计结果预计算和低频接口 HTTP 缓存。

首版暂不引入 Redis、微服务和 Kubernetes。

### 8.5 高并发演进路线

后续商业化高并发按阶段演进：

1. **单机优化**：增强索引、优化查询、Nginx 静态缓存、前端资源压缩、数据预计算。
2. **缓存层**：引入 Redis 缓存热门 bbox 查询、排行榜、指标定义、全区概览和限流计数。
3. **后端横向扩展**：FastAPI 多实例，Nginx 负载均衡，保持后端无状态。
4. **数据库独立与读写分离**：PostGIS 独立部署，只读副本承接查询，Pipeline 写主库。
5. **矢量瓦片与 CDN**：生成 MVT 矢量瓦片，静态瓦片进入对象存储或 CDN。
6. **容器编排与监控体系**：引入 Kubernetes 或托管容器平台，配合 Prometheus、Grafana、日志系统和错误追踪。

## 9. 风险、范围与验收

### 9.1 主要风险

| 风险 | 表现 | 控制方式 |
|---|---|---|
| 数据获取风险 | 思明区 OSM 数据不完整、道路分类不一致 | D5 做数据源审计，保留数据版本和处理日志 |
| 指标解释风险 | 可步行性综合分被误解为绝对评价 | 明确指标含义、权重和适用边界 |
| 性能风险 | 地图一次加载太多道路或网格导致卡顿 | bbox 查询、空间索引、预计算、查询限制 |
| 架构过重风险 | 过早引入 Redis、Kubernetes、微服务导致项目失控 | 首版坚持模块化单体，保留演进边界 |
| 架构过轻风险 | 只做静态网页，缺少数据库和后端，企业感不足 | 坚持 Vue + FastAPI + PostGIS + Pipeline 闭环 |
| 数据版本风险 | 后续结果不知道基于哪份数据 | 所有道路、网格、指标绑定 `data_versions` |
| 审计风险 | GitHub 只有代码，没有过程记录 | 每天提交工作日志、审计报告、PR 或合并记录 |

### 9.2 首版范围

首版包含思明区路网结构分析、可步行性综合评价、500 米网格汇总、道路级与网格级展示、地图交互、指标详情、CSV 导出、数据版本追踪、基础部署设计和 GitHub 审计流程。

首版不包含用户登录、多角色权限、在线编辑权重、实时交通数据、POI/设施可达性、多城市平台化、实时协同、商业级监控平台、Kubernetes 生产集群和矢量瓦片生产链路。

### 9.3 D3 验收标准

D3 完成后应满足：

1. 有清晰的总体架构图和模块边界；
2. 有技术选型理由；
3. 有数据流说明；
4. 有 API 总体设计；
5. 有数据库与数据版本设计；
6. 有前端工作台设计；
7. 有部署与高并发演进设计；
8. 有风险与验收标准；
9. 与 D1 项目章程一致；
10. 覆盖 D2 已确认需求；
11. 不新增 D2 未确认的大型首版功能；
12. 能支撑 D4、D5、D6 继续推进。

## 10. D3 结论

D3 确定本项目采用“离线 Pipeline + PostGIS + FastAPI + Vue 3 GIS 工作台”的模块化单体架构。该架构能够覆盖 D2 需求，符合 D1 范围边界，并为后续商业化高并发演进保留空间。

D3 不要求写业务代码，但必须作为后续模块拆分、数据审计、工程搭建、接口设计、数据库建模和前端工作台开发的架构基线。未经新的用户确认，后续开发不应偏离本架构边界。
